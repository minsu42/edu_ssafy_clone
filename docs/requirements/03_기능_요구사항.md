## 상세 요구사항

### FR-AUTH-LOGIN-001. 로그인

- 상위 요구사항: FR-AUTH
- 우선순위: 상
- 상태: 문서화 완료
- 전제조건: -
- 선행조건: FR-AUTH-LOGOUT-001, FR-AUTH-SESSION-001, FR-AUTH-PROFILE-001, FR-AUTH-CREDENTIAL-001, FR-AUTH-WITHDRAW-001

#### 입력

| 필드 | 타입 | 필수 | 설명 |
|------|------|------|------|
| email | string | Y | 로그인 이메일 |
| password | string | Y | 비밀번호 (평문) |

#### 정상 시나리오

1. 이메일로 사용자 조회 — 존재 확인
2. BCrypt 비밀번호 일치 확인
3. users.status = ACTIVE 확인
4. Access Token(15분), Refresh Token(7일) 발급
5. Refresh Token DB 저장 (기존 토큰 무효화)
6. 토큰 쌍 + 사용자 기본 정보 반환

#### 예외 시나리오

| 조건 | 에러 코드 | HTTP |
|------|-----------|------|
| email 또는 password 필드 누락 | ERR-CMN-003 | 400 |
| 이메일 없음 | ERR-AUTH-001 | 401 |
| 비밀번호 불일치 | ERR-AUTH-001 | 401 |
| status = INACTIVE 또는 WITHDRAWN | ERR-AUTH-002 | 403 |

#### 비즈니스 규칙

- 이메일 미존재와 비밀번호 불일치는 보안상 동일 에러코드 반환 (사용자 열거 방지)
- 로그인 성공 시 기존 Refresh Token 전체 무효화 후 새 토큰 저장

### FR-AUTH-LOGOUT-001. 로그아웃

- 상위 요구사항: FR-AUTH
- 우선순위: 상
- 상태: 문서화 완료
- 전제조건: FR-AUTH-LOGIN-001 (유효한 Access Token)
- 선행조건: -

#### 입력

- `Authorization: Bearer {accessToken}` 헤더

#### 정상 시나리오

1. Access Token 파싱 및 사용자 식별
2. DB에서 해당 사용자의 Refresh Token 삭제
3. 204 No Content 반환

#### 예외 시나리오

| 조건 | 에러 코드 | HTTP |
|------|-----------|------|
| Authorization 헤더 없음 | ERR-CMN-002 | 401 |
| 만료된 Access Token | ERR-AUTH-003 | 401 |
| 유효하지 않은 Access Token | ERR-AUTH-004 | 401 |

#### 비즈니스 규칙

- Refresh Token만 즉시 무효화됨
- 기존 Access Token은 만료 시까지 유효 (JWT Stateless 특성)

---

### FR-AUTH-SESSION-001. 세션 유지 및 로그인 상태 관리

- 상위 요구사항: FR-AUTH
- 우선순위: 상
- 상태: 문서화 완료
- 전제조건: FR-AUTH-LOGIN-001 (유효한 Refresh Token)
- 선행조건: -

#### 입력

| 필드 | 타입 | 필수 | 설명 |
|------|------|------|------|
| refreshToken | string | Y | 로그인/갱신 시 발급된 Refresh Token |

#### 정상 시나리오

1. Refresh Token 서명 및 만료 검증
2. DB에 저장된 토큰과 일치 확인
3. 새 Access Token 발급
4. Refresh Token Rotation: 새 Refresh Token 발급 및 DB 교체
5. 새 토큰 쌍 반환

#### 예외 시나리오

| 조건 | 에러 코드 | HTTP |
|------|-----------|------|
| refreshToken 필드 누락 | ERR-CMN-003 | 400 |
| 만료된 Refresh Token | ERR-AUTH-003 | 401 |
| 서명 검증 실패 | ERR-AUTH-004 | 401 |
| DB에 없는 Refresh Token (로그아웃 후 재사용) | ERR-AUTH-004 | 401 |

#### 비즈니스 규칙

- Refresh Token Rotation 적용: 갱신 시 기존 토큰 폐기, 새 토큰 발급
- 탈취 감지: DB에 없는 토큰으로 갱신 시도 시 해당 사용자의 모든 Refresh Token 무효화

### FR-AUTH-PROFILE-001. 사용자 기본 정보 조회

- 상위 요구사항: FR-AUTH
- 우선순위: 상
- 상태: 문서화 완료
- 전제조건: FR-AUTH-LOGIN-001 (유효한 Access Token)
- 선행조건: FR-AUTH-PROFILE-002

#### 입력

- `Authorization: Bearer {accessToken}` 헤더

#### 정상 시나리오

1. Access Token에서 userId 추출
2. users 테이블 조회
3. profile_file_id가 있으면 files 테이블에서 file_url 조회
4. 프로필 정보 반환

#### 예외 시나리오

| 조건 | 에러 코드 | HTTP |
|------|-----------|------|
| Authorization 헤더 없음 | ERR-CMN-002 | 401 |
| 만료된 Access Token | ERR-AUTH-003 | 401 |
| 유효하지 않은 Access Token | ERR-AUTH-004 | 401 |

#### 비즈니스 규칙

- 응답에 password 필드 미포함
- profile_file_id가 있으면 files.file_url을 profileImageUrl로 반환, 없으면 null

### FR-AUTH-PROFILE-002. 사용자 정보 수정

- 상위 요구사항: FR-AUTH
- 우선순위: 중
- 상태: 문서화 완료
- 전제조건: FR-AUTH-PROFILE-001 (유효한 Access Token)
- 선행조건: -

#### 입력

| 필드 | 타입 | 필수 | 설명 |
|------|------|------|------|
| name | string | N | 변경할 이름 |
| phoneNumber | string | N | 변경할 휴대폰 번호 |
| zipCode | string | N | 변경할 우편번호 |
| address | string | N | 변경할 기본 주소 |
| addressDetail | string | N | 변경할 상세 주소 |
| profileFileId | long | N | 변경할 프로필 이미지 파일 ID |

#### 정상 시나리오

1. Access Token에서 userId 추출
2. 제공된 필드만 부분 업데이트 (PATCH 의미론)
3. 수정된 프로필 정보 반환

#### 예외 시나리오

| 조건 | 에러 코드 | HTTP |
|------|-----------|------|
| Authorization 헤더 없음 | ERR-CMN-002 | 401 |
| 만료된 Access Token | ERR-AUTH-003 | 401 |
| 존재하지 않는 profileFileId | ERR-CMN-001 | 404 |

#### 비즈니스 규칙

- 수정 불가 필드: email, role, status, studentNo, generation, region, classNo (요청에 포함되어도 무시)
- 제공된 필드만 업데이트 (null 전송 시 무시)

### FR-AUTH-CREDENTIAL-001. 비밀번호 변경 / 로그인 정보 변경 기능

- 상위 요구사항: FR-AUTH
- 우선순위: 중
- 상태: 문서화 완료
- 전제조건: FR-AUTH-LOGIN-001 (유효한 Access Token)
- 선행조건: -

#### 입력

| 필드 | 타입 | 필수 | 설명 |
|------|------|------|------|
| currentPassword | string | Y | 현재 비밀번호 |
| newPassword | string | Y | 새 비밀번호 |

#### 정상 시나리오

1. currentPassword와 DB의 BCrypt 해시 일치 확인
2. newPassword 유효성 검증
3. newPassword BCrypt 인코딩 후 users.password 저장
4. password_change_histories에 이력 기록 (ip_address, user_agent 포함)
5. 해당 사용자의 모든 Refresh Token 무효화
6. 204 No Content 반환

#### 예외 시나리오

| 조건 | 에러 코드 | HTTP |
|------|-----------|------|
| Authorization 헤더 없음 | ERR-CMN-002 | 401 |
| 만료된 Access Token | ERR-AUTH-003 | 401 |
| currentPassword 또는 newPassword 필드 누락 | ERR-CMN-003 | 400 |
| 현재 비밀번호 불일치 | ERR-AUTH-005 | 400 |
| 새 비밀번호 유효성 실패 | ERR-CMN-003 | 400 |

#### 비즈니스 규칙

- 새 비밀번호: 8자 이상, 영문 + 숫자 + 특수문자(`!@#$%^&*`) 각 1자 이상 포함
- 비밀번호 변경 후 모든 Refresh Token 무효화 → 재로그인 필요
- 변경 이력은 password_change_histories에 기록 (감사 목적)

### FR-AUTH-STATUS-001. 계정 상태 검증

- 상위 요구사항: FR-AUTH
- 우선순위: 상
- 상태: 문서화 완료
- 전제조건: -
- 선행조건: -

#### 정상 시나리오

- JwtAuthFilter에서 모든 인증 필요 엔드포인트 요청 시 users.status 확인
- status = ACTIVE → 정상 처리 통과

#### 예외 시나리오

| 조건 | 에러 코드 | HTTP |
|------|-----------|------|
| status = INACTIVE | ERR-AUTH-002 | 403 |
| status = WITHDRAWN | ERR-AUTH-002 | 403 |

#### 비즈니스 규칙

- 별도 API 없음 — JwtAuthFilter 내부 처리
- 계정 상태 변경 후 기존 Access Token 만료 전(최대 15분)까지 접근 가능 (JWT Stateless 특성)

### FR-AUTH-WITHDRAW-001. 회원 탈퇴 또는 계정 비활성화

- 상위 요구사항: FR-AUTH
- 우선순위: 하
- 상태: 문서화 완료
- 전제조건: FR-AUTH-LOGIN-001 (유효한 Access Token)
- 선행조건: -

#### 입력

- `Authorization: Bearer {accessToken}` 헤더

#### 정상 시나리오

1. Access Token에서 userId 추출
2. users.status = WITHDRAWN 변경
3. 해당 사용자의 모든 Refresh Token 삭제
4. 204 No Content 반환

#### 예외 시나리오

| 조건 | 에러 코드 | HTTP |
|------|-----------|------|
| Authorization 헤더 없음 | ERR-CMN-002 | 401 |
| 만료된 Access Token | ERR-AUTH-003 | 401 |

#### 비즈니스 규칙

- 물리 삭제 없음 — users.status만 WITHDRAWN으로 변경
- 탈퇴 후 동일 이메일로 재로그인 시도 → ERR-AUTH-002 반환
